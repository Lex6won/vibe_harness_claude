# 예외 정책 — 금지 vs 예외승인 (우회 방지 설계)

스킬이 "안 됩니다"만 반복하면 사용자는 스킬을 끄고 우회한다.
**금지(절대 불가)**와 **예외승인(절차 있음)**을 구분하고, 예외는 *신청 양식을 자동 작성*한다.

## 금지 (불변 — 차단 + 대체 안내)

| 항목 | 대체 |
|---|---|
| 외부 LLM에 개인정보 입력 | 가명·더미 데이터, 내부 sLLM |
| 실데이터로 개발 | 테스트 데이터 |
| 이용자가 서버·이미지 직접 반출 | 소스+Dockerfile만, 내부 빌드 |
| 인증·암호화 직접 구현 | Keycloak / 표준 라이브러리 |

## 예외승인 가능 (절차 있음 — 05_예외신청서 자동 작성)

| 항목 | 승인 주체 |
|---|---|
| 신규(미승인) 패키지 | 운영단 보안 담당 (3일 내 검증·등재) |
| 외부 API 연계 | 팀단 (망연계 기준) |
| 개인정보 처리 서비스 | 개인정보보호 부서 협의 + 팀단 보고 |
| DMZ/대민 배포 | 팀단 건별 의결 |
| gvskb 룰 예외 | 운영단 보안 담당 (사유·승인자·만료 필수, `.gvskb-exceptions.yaml`) |

## 예외신청 자동 작성 (deployment-clerk)

`05_예외신청서`에 자동 채움: 사유 / 대체수단 검토 / 영향 범위 / 담당 / 승인 주체 / 예상 기간.
manifest.exceptions에 기록(item·reason·approved_by·expires — 3필드 필수).

## 원칙
- 금지 항목도 반드시 대체 방법을 함께 제시.
- 예외는 "안 됨"이 아니라 "이렇게 신청하면 됨"으로 안내.
